## Краткая сводная таблица

| Метод | Формула | Сложность | Когда применять |
|---|---|---|---|
| **MoSCoW** | Must / Should / Could / Won't | Простая | Стейкхолдер-митинги, MVP-скоупинг, фиксированный бюджет/срок |
| **ICE** | Impact × Confidence × Ease (1-10) | Низкая | Discovery, гипотезы, ранний продукт, нет данных о Reach |
| **RICE** | (Reach × Impact × Confidence) / Effort | Средняя | Зрелый продукт с известной аудиторией |
| **WSJF** | Cost of Delay / Job Size | Высокая | SAFe, портфельная приоритизация ART |
| **Kano** | Basic / Performance / Excitement / Indifferent | Опросная | Анализ восприятия фич |
| **Story Mapping** | Карта user journey + срезы релизов | Визуальная | Скоупинг релизов, MVP, рефакторинг роадмапа |
| **Value vs Effort** | 2×2 матрица | Самая простая | Быстрые рабочие сессии |
| **Cost of Delay** | $ потерь от задержки | Финансовая | Продуктовые команды с измеримой выручкой на фичу |
| **Buy a Feature** | Игровая, виртуальный бюджет | Игровая | Customer-development сессии |

**Главный практический выбор:**
- Reach знаешь? → RICE
- Не знаешь? → ICE
- Со стейкхолдерами обсудить scope? → MoSCoW
- Внутри SAFe? → WSJF
- Опрашиваешь клиентов? → Kano

## ICE

**Формула:** `Impact × Confidence × Ease`, шкала каждого 1-10.

- **Impact** — насколько улучшает ключевые метрики (1-10)
- **Confidence** — насколько уверены в оценке Impact: интуиция → 1-3, аналитика → 4-6, A/B-тесты → 7-10
- **Ease** — обратная сложности. 10 = легко, 1 = очень тяжело

**Происхождение.** Sean Ellis (GrowthHackers).

**Плюсы:** дешево, быстро, без данных. Confidence явно учитывает «не знаем».

**Минусы:** Reach неявно. Большие охваты vs нишевые не различимы.

**Применять:** Стартапы pre-PMF / ранний post-PMF, discovery-команды, гипотезы.

## RICE

**Формула:** `(Reach × Impact × Confidence) / Effort`.

- **Reach** — сколько пользователей затронет за период (число)
- **Impact** — 0.25 / 0.5 / 1 / 2 / 3 (massive)
- **Confidence** — 50% / 80% / 100%
- **Effort** — человеко-месяцы

**Происхождение.** Intercom.

**Плюсы:** учитывает охват. Effort в знаменателе — экономика прозрачна.

**Минусы:** требует Reach. Для нового сегмента не работает.

**Применять:** Зрелый продукт, есть данные о Reach, идёт расстановка стратегических ставок.

## MoSCoW

**Что это.** Категории:
- **Must have** — без этого релиз нет смысла
- **Should have** — важно, но без этого можно жить
- **Could have** — полезно, но не критично
- **Won't have** — не сейчас

**Происхождение.** Dai Clegg, Oracle, 1994.

**Плюсы:** простой, понятный нетехническим стейкхолдерам. Хорош для переговоров «что в MVP».

**Минусы:** часто «всё Must». Нет численной шкалы внутри Must.

**Применять:** Скоупинг MVP, переговоры с клиентом/стейкхолдером с фиксированным бюджетом/сроком.

## WSJF (Weighted Shortest Job First)

**Формула:** `Cost of Delay / Job Size`. Cost of Delay = User-Business Value + Time Criticality + Risk Reduction / Opportunity Enablement. Все компоненты — Fibonacci (1, 2, 3, 5, 8, 13, 20).

**Происхождение.** SAFe (Dean Leffingwell). На основе lean economics.

**Плюсы:** оптимизирует поток ценности через ограниченные ресурсы.

**Минусы:** тяжёлый, требует команду >50 чел и обученного RTE. Без культуры lean — превращается в формальность.

**Применять:** Только внутри SAFe. Не пытаться внедрить отдельно.

## Kano Model

**4 категории фич:**
- **Basic / Must-be** — без них недовольство (вода в кране)
- **Performance** — линейная зависимость удовлетворения
- **Excitement / Delighters** — wow-эффект
- **Indifferent** — не влияет

**Происхождение.** Noriaki Kano, 1980-е.

**Метод:** опрос «как вы отнесётесь, если фича будет / если её не будет».

**Применять:** Customer research, анализ восприятия конкретных фич. Не для повседневной приоритизации.

## Story Mapping

**Что это.** Двумерная карта: горизонталь — user journey (от первого касания до результата), вертикаль — варианты решения. Срезы карты = релизы.

**Происхождение.** Jeff Patton, *User Story Mapping* (2014).

**Применять:** Скоупинг релизов, рефакторинг роадмапа, MVP-выбор.

## Value vs Effort

**Что это.** 2×2 матрица: ось X — Effort (low/high), ось Y — Value (low/high). Quick wins (high value / low effort) делать первыми.

**Применять:** Быстрые сессии команды, когда нет времени на ICE/RICE.

## Cost of Delay

**Что это.** Денежная оценка: сколько теряем за каждую неделю задержки фичи. Помогает оправдать срочность.

**Применять:** Зрелые продукты с измеримой выручкой на фичу. Sales-driven компании.

## Buy a Feature

**Что это.** Игровой метод: стейкхолдеры получают виртуальный бюджет, «покупают» фичи. Каждая фича имеет цену = effort.

**Применять:** Customer-development сессии, переговоры с клиентами, прояснение willingness to pay.

## Опыт Avito и практические хаки

Команда Avito (2021) переходила с WSJF/RICE на ICE и описала это публично на Habr (статья 571378). Их выводы:

1. **ICE > RICE/WSJF при малом объёме данных.** В discovery-фазе Reach неизвестен — ICE через Confidence явно это учитывает.

2. **Шкалы Impact (1-10):** баллы складываются за признаки. «Улучшает UX» = 2, «влияет на OKR» = 2, «открывает новый сегмент» = 3 и т.д.

3. **Ease (1-10) — обратная Story Points.** 10 = ≤2 SP, 1 = 21+ SP (несколько спринтов).

4. **Хак: задачи ≤2 SP без скоринга.** Overhead скоринга больше выигрыша. Просто делать.

5. **Хак: «зелёная задача» в каждый спринт.** Морально-командная, даже если по ICE проигрывает.

6. **PSBR ритуал.** Product Score Backlog Refinement — 30 минут раз в неделю, ~10 задач за встречу. Регулярность важнее идеальности скоринга.

7. **Инструмент.** Jira + Issue Score plugin ($250/год). Позволяет формулу + сортировку. Hygger пробовали — отказались из-за дублирования.

**Применять эти хаки для команд 6-15 чел post-PMF.** Для более крупных — добавлять Reach (RICE), для более ранних — оставаться на ICE без хаков.
